# Technical Design: Stage 1 Offline Face Auth Backend

## Backend Overview

Build the backend in Go 1.24 using Gin and a pluggable persistence store. The hackathon/default runtime uses JSON files; MongoDB remains available for deployments that set `STORE_BACKEND=mongo`.

The backend owns:

- Tenant creation and configuration.
- User onboarding and face embedding enrollment.
- Client/device onboarding.
- Stage 1 app login response for device registration.
- Offline profile generation.
- Bulk offline auth event sync.
- Purge acknowledgement.
- Admin/demo read APIs.

The backend does not own:

- Face recognition.
- Liveness detection.
- Mobile model execution.
- JWT authentication in Stage 1.

## Core Domain Flow

1. Admin creates a tenant.
2. Admin creates a user under that tenant.
3. User downloads the app and logs in.
4. Stage 1 login returns `tenantId` and `userId`.
5. In Stage 2, login returns a JWT and backend decodes `tenantId` and `userId` from JWT claims.
6. After login, the app shows a register-device button.
7. The register-device action calls create client with `tenantId`, `userId`, and device metadata.
8. Backend generates a globally unique opaque `clientId`.
9. App stores `clientId`.
10. App sends only `clientId` for post-registration runtime APIs.
11. Backend resolves tenant and user from `clientId`.

## Storage Collections

In file mode, each collection is stored as a JSON file in `FILE_STORE_DIR`. In Mongo mode, the same logical collections map to MongoDB collections.

### tenants

Stores organization-level model and liveness configuration.

Fields:

- `id`
- `name`
- `status`
- `configs`
- `createdAt`
- `updatedAt`

Supported config keys:

- `MODEL_CONFIG`
- `LIVENESS_CONFIG`

`MODEL_CONFIG` fields:

- `modelVersion`
- `faceThreshold`
- `livenessThreshold`
- `embeddingDimension`
- `modelChecksum`
- `active`

`LIVENESS_CONFIG` fields:

- `challengeTypes`
- `active`

### users

Stores tenant-scoped user profile and enrolled embeddings.

Fields:

- `id`
- `tenantId`
- `employeeId`
- `name`
- `role`
- `status`
- `embeddings`
- `createdAt`
- `updatedAt`

Rules:

- `employeeId` is unique only inside a tenant.
- Multiple embedding vectors are allowed.
- Embedding vector length must match tenant `MODEL_CONFIG.embeddingDimension`.
- Embeddings stay embedded in the user document for Stage 1.

### clients

Stores one registered app/device installation mapped to one user.

Fields:

- `id`
- `clientId`
- `tenantId`
- `userId`
- `deviceType`
- `deviceName`
- `platform`
- `appVersion`
- `imei`
- `status`
- `activatedAt`
- `deactivatedAt`
- `createdAt`
- `updatedAt`

Rules:

- `clientId` is globally unique.
- `clientId` is generated by the backend.
- Client ownership cannot be reassigned.
- To move a device to another user, deactivate the old client and create a new client.

### auth_events

Stores offline verification events synced by the app.

Fields:

- `id`
- `tenantId`
- `userId`
- `clientId`
- `eventId`
- `result`
- `failureReason`
- `faceScore`
- `livenessScore`
- `challengeTypes`
- `latencyMs`
- `embedding`
- `capturedAt`
- `receivedAt`
- `purgeStatus`

Rules:

- `eventId` is generated by the mobile app.
- Idempotency is enforced by `(clientId, eventId)`.
- Event embeddings are stored in Stage 1 because the current plan requires it.

## API Contracts

### Tenant APIs

- `POST /api/tenants`
- `GET /api/tenants`
- `GET /api/tenants/{tenantId}`
- `PUT /api/tenants/{tenantId}`
- `DELETE /api/tenants/{tenantId}`

Delete is a soft delete.

### User APIs

- `POST /api/tenants/{tenantId}/users`
- `GET /api/tenants/{tenantId}/users`
- `GET /api/tenants/{tenantId}/users/{userId}`
- `PUT /api/tenants/{tenantId}/users/{userId}`
- `DELETE /api/tenants/{tenantId}/users/{userId}`

User create/update includes multiple face embeddings.

### Client APIs

- `POST /api/clients`
- `GET /api/tenants/{tenantId}/clients`
- `GET /api/tenants/{tenantId}/clients/{clientId}`
- `PUT /api/tenants/{tenantId}/clients/{clientId}`
- `DELETE /api/tenants/{tenantId}/clients/{clientId}`

Create client is called by the app after login. In Stage 1 the request body contains `tenantId`, `userId`, and device metadata. In Stage 2 the backend should read tenant/user from JWT claims and either remove request body ids or validate they match the token.

Client update can change metadata/status only. It must not change `tenantId` or `userId`.

### Login API

- `POST /api/login`

Stage 1 login is a simple app integration endpoint. It returns `tenantId` and `userId` for the logged-in user so the app can call create client. Stage 2 replaces this response contract with JWT-based auth where `tenantId` and `userId` are decoded from token claims.

### Runtime Mobile APIs

- `GET /api/clients/{clientId}/offline-profile`
- `POST /api/clients/{clientId}/sync/events`
- `POST /api/clients/{clientId}/sync/purge-ack`

Runtime APIs after client registration must not require tenantId or userId from the app.

## Offline Profile Response

The offline profile includes:

- `clientId`
- `tenantId`
- `userId`
- `employeeId`
- `userName`
- `modelConfig`
- `livenessConfig`
- `embeddings`
- `validUntil`
- `signature`

For Stage 1, `signature` is present but nullable. Keep a signer interface so Stage 2 can add real signing without changing the response shape.

## Deep Modules

### Client Resolver

Input:

- `clientId`

Output:

- client
- tenant
- user
- resolved active/inactive state

Responsibilities:

- Find client by `clientId`.
- Find tenant and user by stored ids.
- Confirm tenant/user/client consistency.
- Decide whether profile generation is allowed.
- Provide a stable interface for profile, sync, purge, and future JWT-aware flows.

### Login Context

Stage 1 login returns explicit ids:

- `tenantId`
- `userId`

Stage 2 login returns JWT. Device registration should then derive tenant/user from decoded JWT claims, not trust app-supplied ids.

### Profile Builder

Input:

- resolved client context

Output:

- offline profile response

Responsibilities:

- Combine user info, embeddings, tenant model config, tenant liveness config, validity, and signature.
- Use signer interface.
- Keep profile response stable for mobile.

### Config Validator

Responsibilities:

- Validate model thresholds.
- Validate liveness challenge types.
- Validate embedding dimension.
- Validate user and event embedding vectors against tenant model config.

### Event Sync Service

Responsibilities:

- Resolve client context.
- Validate batch size.
- Validate events.
- Enforce idempotency.
- Store accepted events.
- Return accepted, duplicate, and rejected event ids.

### Purge Service

Responsibilities:

- Resolve client context.
- Mark known events as purged.
- Keep purge acknowledgement idempotent.
- Return unknown event ids separately.

## Stage 2 Readiness

Stage 1 has no auth, but the code should keep these extension points:

- Stage 1 login exposes `tenantId` and `userId`; Stage 2 replaces this with JWT claims.
- Central tenant/user/client resolution through client resolver after device registration.
- Signer interface for offline profile signature.
- Handler/service separation so JWT middleware can be added later.
- No direct reliance on mobile-supplied tenantId or userId for post-registration runtime APIs.
